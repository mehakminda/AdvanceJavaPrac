Strong and weak refrence:

Striong: Person p=new Person();
Weak:  WeakPerson<Person> p = new WeakPerson<Person>(new Person())
->in above statement we are creating a wrapper around a Person object using a class called WeakPerson
WeakPerson<Person> means you're using a generic class WeakPerson with Person as the type parameter

to above we can also pass : WeakPerson<Apple> p = new WeakPerson<Apple>(new Apple());
Because The WeakPerson class is generic and can accept any type — not just Person.

WeakPerson<Person> p = new WeakPerson<Person>(); ->
Creates an empty wrapper, possibly to be set later.
this means creating a new instance of the generic class WeakPerson without passing a Person object to its constructor

What is meant by creating a wrapper?
In programming, creating a wrapper means building a class or object that encapsulates another object — like putting it inside a container — to add extra functionality, control access, or modify behavior.

Think of a wrapper like a gift box:

-The actual gift is the object (e.g., Person, Apple).
-The box (wrapper) holds the gift and might add a label, protection, or instructions.
-You can interact with the box to get the gift, but the box might also do other things (like log access or delay opening).

if it was : WeakPerson<Apple> p = new WeakPerson<>(new Apple());

We are wrapping the Apple object inside a WeakPerson object. This wrapper might:
-Hold a weak reference to the Apple object.
-Provide methods like get() to access it.
-Allow the object to be garbage collected if no strong references exist.

Usecases :
Memory management:	Using WeakReference to avoid memory leaks.
Adding behavior: Logging every time an object is accessed.
Security:	Restricting access to sensitive data.

Stack Memory:

- Store Temporary Variables and separate memory block for methods.
- Store Primitive data types
- Store Reference of the heap objects
	. Strong reference
	· Weak reference
		Soft reference
- Each thread has its own Stack memory.
- Variables within a SCOPE is only visible and as soon as any variable goes out of the SCOPE, it get deleted from the Stack (in LIFO order)
- When Stack memory goes full, its throws "java.lang.StackOverflowError"


Heap Memory:

Store Objects.
There is no order of allocating the memory.
Garbage Collector is used to delete the unreferenced objects from the heap.
 ->Mark and sweep Algorithm
 ->Types of GC:
	Single GC
	Parallel GC
	CMS (concurrent Mark Sweep)
	G1
Heap memory is shared with all the threads.
Heap also contains the String Pool.
When Heap memory goes full, its throws "java.lang.OutofMemoryError"
Heap memory is further divided into:
	->Young Generation (minor GC happens here)
		Eden
		Survivor
Old/Tenured Generation (major GC happens here)
Permanent Generation
